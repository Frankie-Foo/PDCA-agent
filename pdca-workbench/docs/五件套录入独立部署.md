# 门店五件套录入（/walkin-submit）· 独立部署

## 背景

2026-07-15 发生过一次事故：对外域名 `pdca-workbench.vertu.cn`（经销商用来提交五件套录入）
返回 502，而内部域名 `pdca-workbench-teams.vertu.cn`（PDCA 工作台，内部人员用）正常。排查
发现两个域名解析到同一台服务器，是 Caddy 反代层面对外域名这条路由指向的后端没起来——根因
是当时新增的整套 CI/CD 自动部署管道（`deploy_remote_docker.ps1`、GHCR 镜像发布、健康检查）
只覆盖了 `-teams` 域名，仓库里从未出现过对外域名的任何部署配置。内部工作台的部署管道改动，
连带炸了外部经销商入口。

为避免同类事故再次发生，五件套录入现在拥有**完全独立的部署单元**——独立容器、独立端口、
独立 Caddy 路由、独立部署脚本。以后无论怎么折腾 PDCA 工作台（内部看板）的部署管道，都不会
再影响到经销商使用的对外入口。

## 两套服务对照

| | PDCA 工作台（内部） | 五件套录入门户（对外） |
|---|---|---|
| 容器名 | `pdca-workbench` | `pdca-walkin-portal` |
| 主机端口 | `127.0.0.1:8768→8767` | `127.0.0.1:8769→8767` |
| 域名 | `pdca-workbench-teams.vertu.cn` | `pdca-workbench.vertu.cn` |
| Compose 文件 | `docker-compose.yml`（服务名 `pdca-app`） | `docker-compose.walkin.yml`（服务名 `pdca-walkin-app`） |
| 部署脚本 | `scripts/deploy_remote_docker.ps1` | `scripts/deploy_remote_docker_walkin.ps1` |
| 敏感配置 | `.env` | `.env.walkin` |
| 非敏感默认配置 | `env.docker` | `walkin.env.docker` |
| 镜像 | `ghcr.io/frankie-foo/pdca-workbench:$Sha` | 同一个镜像，同一个 SHA（不单独构建） |
| 数据库 | 同一个生产 Postgres（`10.100.0.176:5432/pdca`） | 同一个生产 Postgres |
| SECRET_KEY | 各自独立，互不通用（JWT/cookie 不互通） | 各自独立 |
| PDCA_AUTH_MODE | `hybrid`（本地 + VPS） | `local`（经销商账号不需要 VPS/vertu-cli SSO） |
| PDCA_REQUIRE_VERTU | `1` | `0` |
| PDCA_SCHEDULER_ENABLED | `1`（每日 VPS 同步/物流追踪/KPI 刷新） | `0`（避免两个容器各跑一遍同样的定时任务） |
| 备份目录 | `backups/` | `backups-walkin/`（各自独立命名，内容其实是同一个库的备份） |
| 数据目录 | `data/` | `data-walkin/` |
| `/mvp`、`/repo` 挂载 | 需要（经营首页等模块用） | 不需要（部署手册已确认五件套录入不依赖 MVP 数据，因此这个部署脚本也不做 git archive/per-SHA release 目录暂存） |

两个容器共用同一份代码镜像和同一个生产库，但作为两个完全独立的进程运行——重启/重新部署
任意一个都不会影响另一个。

## 部署步骤

### 1. 准备 `.env.walkin`

在 `pdca-workbench/` 目录下，复制 `.env.walkin.example` 为 `.env.walkin`（未跟踪，不要提交），
填入：

- `PDCA_WALKIN_SECRET_KEY`：独立生成一个 32 位以上随机密钥（不要和主服务的 `PDCA_SECRET_KEY`
  共用），生成方式：
  ```bash
  python -c "import secrets; print(secrets.token_urlsafe(32))"
  ```
- `PDCA_DATABASE_URL`：可以直接沿用主服务同一个生产库连接串，已有 `users`/`dealer_stores`/
  `walkin_daily_reports` 表和经销商账号，不需要建新库或迁移数据。

### 2. 在内网能连上 `10.100.0.176` 的机器上执行部署

```powershell
cd D:\经销商PDCA\pdca-workbench
powershell -ExecutionPolicy Bypass -File .\scripts\deploy_remote_docker_walkin.ps1
```

脚本会自动：挑选最新一次 `main` 分支绿灯的 CI 镜像 → 备份 Postgres → 起
`pdca-walkin-portal` 容器 → 等待健康检查 → 探测公网 `$PublicUrl/health`、`/login`、
`/walkin-submit` 三个端点 → 任一失败自动回滚到上一个镜像。

首次执行前，先确认端口 8769 没被占用、旧的 502 之前指向的到底是哪个进程：

```bash
docker -H tcp://10.100.0.176:2375 ps -a
```

如果发现一个已经停止/僵死的旧进程占着 `pdca-workbench.vertu.cn` 原本该用的端口或名字，
先清理掉，避免和新容器冲突。

### 3. 在服务器上添加 Caddy vhost

仓库里没有 Caddy 配置文件（不受版本控制），需要登录服务器直接找到 Caddy 配置（常见路径
`/etc/caddy/Caddyfile`，或者由某个 docker-compose 管理的 Caddy 容器里）。照抄现有
`pdca-workbench-teams.vertu.cn` 那个 vhost block 的证书/格式写法，新增一段，只改域名和端口：

```
pdca-workbench.vertu.cn {
    reverse_proxy 127.0.0.1:8769
}
```

加完后 reload（取决于服务器上 Caddy 的运行方式）：

```bash
caddy reload
# 或
systemctl reload caddy
```

### 4. 端到端验证

用一个经销商测试账号访问 `https://pdca-workbench.vertu.cn/walkin-submit`：登录、提交一条
五件套数据、刷新看到历史记录。最终判据——把 `pdca-workbench` 容器停掉，确认
`pdca-workbench.vertu.cn` 完全不受影响，这就是隔离生效的证明：

```bash
docker -H tcp://10.100.0.176:2375 stop pdca-workbench
curl -I https://pdca-workbench.vertu.cn/walkin-submit   # 应该仍是 200/307，不是 502
docker -H tcp://10.100.0.176:2375 start pdca-workbench
```

## 常用命令

```bash
# 独立容器日志
docker -H tcp://10.100.0.176:2375 logs -f pdca-walkin-portal

# 独立容器健康检查
curl http://127.0.0.1:8769/health   # 需在服务器本机执行

# 本地起独立容器做冒烟测试（不需要连内网）
cd pdca-workbench
docker compose -f docker-compose.walkin.yml up -d --build
curl http://127.0.0.1:8769/health
curl -L http://127.0.0.1:8769/walkin-submit
docker compose -f docker-compose.walkin.yml down
```

## 后续可选优化

当前是「同一套代码镜像，起第二个独立容器」（方案A）——不改应用代码，靠独立容器/端口/
Caddy vhost/部署脚本实现进程级隔离，风险低、当天就能落地。

如果以后想进一步收紧镜像体积和攻击面，可以考虑方案B：把 `app/main.py` 拆出一个只挂载
`auth`/`walkin`/`export`/`pages` 相关路由的精简版本，Dockerfile 也可以跳过
`playwright install`（walkin 路由不依赖它）。这属于锦上添花，不是这次隔离要解决的问题，
本次没有做。
